## Security model (do not weaken)

- Real boundary = Supabase **RLS + `SECURITY DEFINER` RPCs**. Client code is not a security layer вЂ” the browser holds the publishable anon key + the user's own JWT, so it can call Supabase directly (bypassing the Next.js server and every Server Action guard) via browser devtools. **`requireAdmin()` checks in Server Actions are defense-in-depth, not the boundary вЂ” RLS is.**
- Only `NEXT_PUBLIC_SUPABASE_URL` + `NEXT_PUBLIC_SUPABASE_ANON_KEY` (publishable) reach the browser. **Never** put the service_role key client-side.
- Money (net revenue, VAT, expense totals, payslip figures) is **recomputed server-side** (Server Actions / `lib/payslip.js` `computePayslip()`) вЂ” never trust client-sent totals.
- Auth: login is a Server Action (also checks `profiles.is_active`); `middleware.js` guards every route; pages use `requireSession()` (`lib/session.js`, also re-checks `is_active` and signs out disabled accounts).
- **`lib/perms.js` role-visibility matrix (`role_perms`) is UI-only** вЂ” it hides nav tabs, it does not restrict data access. Do not treat it as an access-control mechanism; RLS is what actually gates reads/writes.
- **Run `harden_security.sql` (repo root) in Supabase SQL Editor if not already applied вЂ” it's idempotent, safe to re-run any number of times.** Fixes, in order: (1) profiles self-role-escalation вЂ” `profiles: update own` had no column restriction, so any logged-in user could set their own `role` to `'admin'` via a direct Supabase call; now a trigger blocks self-changes to `role`/`is_active` unless already admin; (2) `business_config` writes tiered by key вЂ” `role_perms`/`opex_defaults` admin-only, `emp_details`+`form50_payees` admin+co-admin, everything else (`biz_info`, `emp_pay_history`, etc.) unchanged/open; (3) `audit_log` reads restricted to admin; (4) `sales_daily`/`expenses` UPDATE restricted to admin/co-admin/manager (excludes `staff` вЂ” staff can still INSERT new entries, just not edit existing ones); (5) `business_config` changes now audit-logged too (see below).
- **вљ пёЏ RLS gotcha discovered 2026-07-19: Postgres RLS `UPDATE ... USING` (no `WITH CHECK`) fails silently, not loudly.** If a row already exists and the caller's role fails the `USING` clause, the UPDATE matches **zero rows** вЂ” no error, `error` stays `null`. `.upsert()` calls that don't `.select()` can't even detect this, so a blocked save just looks like "the button does nothing." All `business_config` upserts now go through `lib/config-store.js` `upsertBusinessConfig()`, which `.select('key')` and treats an empty result as a permission error. **Any new `business_config` write must use this helper**, not a bare `.upsert()`. Also: `emp_pay_history`'s RLS tier must always match whatever role can trigger `saveOpexAction` (currently any authenticated user) вЂ” restricting it tighter than that silently breaks pay-history recording for non-admin OPEX saves, which is exactly what happened before this fix. UI now also disables (not just hides) `emp_details`/`form50_payees` edit fields for roles that can't save them, so the mismatch fails visibly and early instead of silently at submit time.
- `audit_log` (with `old_data`/`new_data`, trigger-populated, `SECURITY DEFINER` so it can't be forged by clients) tracks every INSERT/UPDATE/DELETE on `sales_daily`, `expenses`, and (since 2026-07-20) `business_config`. Viewable at **`/admin/audit`** (admin-only вЂ” enforced both at the page level via `redirect('/dashboard')` and at the RLS level) вЂ” filterable by table/action, shows who + when + a diff of changed fields per row; useful for detecting/investigating tampering after the fact. **`business_config`'s primary key is `key` (text), not `id`, so it needed a separate trigger function `fn_audit_log_config()`** (the shared `fn_audit_log()` used by sales/expenses references `new.id`/`old.id`, which doesn't exist on that table) вЂ” `record_id` is stored `null` for config rows; which config entry changed is visible from `old_data`/`new_data`'s `key` field instead, and the audit UI surfaces it directly in the row summary (`app/admin/audit/audit-row.js`).
- **Form 50 аё—аё§аёґ payee editing (`app/opex/form50.js`) moved from admin-only to admin+co-admin on 2026-07-20** вЂ” matches `emp_details`'s existing tier. Prop renamed `isAdmin` в†’ `canEdit` throughout that file to reflect this (still admin-only in the RLS sense for `role_perms`/`opex_defaults`, just not for `form50_payees` anymore).
- **Numeric input sanitization (`lib/format.js`, added 2026-07-20):** `sanitizeNumberString()` strips leading zeros live while typing (`"0123"` в†’ `"123"`, but keeps `"0"` and `"0.5"` intact) and strips any character that isn't a digit or a decimal point вЂ” this blocks `e`/`+`/`-` which native `<input type="number">` otherwise accepts (typing `"1e5"` would silently become a huge number). Applied to every money/quantity input in the app, either centrally (wherever a shared `set(k)`-style handler exists вЂ” `sales-form.js`, `opex-form.js`'s `Row`/`SlipField` components) or at the individual call site where the handler is generic and shared with non-numeric fields (`expense-form.js`, `expense-list.js`, `settings-form.js`, `opex-defaults.js`). **Do not add a raw numeric `onChange={(e) => set(k, e.target.value)}` anywhere вЂ” always run the value through `sanitizeNumberString()` first.**
  - `stripDigits()` вЂ” for genuine person-name fields (employee `fullname`/`lastname`/`account_holder`, Form 50 payee `name`, admin user `full_name`/`nickname`) so a name field can't end up containing digits.
  - `digitsOnly()` вЂ” for fields that must be numeric but where a leading zero is meaningful and must NOT be stripped (Thai national ID `id_card`, bank `account_no`, Form 50 `taxid`) вЂ” distinct from `sanitizeNumberString()`, which is only for money/quantity values where leading zeros are meaningless.
  - Deliberately **not** applied to item/product/supplier name fields (expense `item_name`, `supplier`) вЂ” those legitimately contain digits (e.g. "аё™аёЎаёЄаё” UHT 1L", "а№ЃаёЃа№‰аё§ 12oz").
- Password minimum length is `MIN_PASSWORD_LENGTH` in `lib/auth-policy.js` (currently 8), enforced in `createUserAction`/`resetPasswordAction`. **Also set the same (or higher) minimum in Supabase Dashboard в†’ Authentication в†’ Providers в†’ Email в†’ Minimum password length** вЂ” the app-side check alone doesn't constrain Supabase Auth itself.

## Performance

- **`lib/session.js`'s `requireSession()` uses `getSession()`, not `getUser()` вЂ” deliberately, not an oversight.** `getUser()` makes a real network round-trip to Supabase's Auth server to re-verify the JWT; `getSession()` just reads/decodes it from cookies locally. `middleware.js` already calls `getUser()` (the network-verified, authoritative check) on every request before any page or Server Action runs, refreshing cookies if needed вЂ” by the time `requireSession()` runs, the cookies it reads via `lib/supabase/server.js` already reflect that verified session, so re-verifying with a second `getUser()` call was pure added latency (two Supabase Auth round-trips per page load instead of one) with no real security benefit. **If you ever see `requireSession()` using `getUser()` again, that's a regression вЂ” change it back to `getSession()`.** Note this optimization was deliberately *not* extended to the many standalone `supabase.auth.getUser()` calls inside individual Server Actions (`app/*/actions.js`) вЂ” those aren't on the page-load critical path (they run on form submit, not navigation) and touching auth checks across many mutation endpoints at once was judged higher-risk for lower payoff; revisit only if Server Action latency specifically becomes a complaint.
- **Icon font is self-hosted, not loaded from a CDN.** `app/layout.js` used to `<link>` `@tabler/icons-webfont` from `cdn.jsdelivr.net` вЂ” a render-blocking external request (separate DNS + TLS + download) on every single page load, on top of whatever the VPS itself serves. Now installed as a real dependency and imported directly (`import '@tabler/icons-webfont/dist/tabler-icons.min.css'` in `app/layout.js`), so Next.js bundles and self-hosts it under `/_next/static/media/` like any other static asset вЂ” same reasoning as the Prompt font switch to `next/font/google` (self-hosted, no third-party runtime request). **Never add a `<link>` to an external CDN for a font/icon set again вЂ” install it and import it instead.**

## Domain formulas (keep identical to legacy)

- Delivery GP rates: shopee 0.3424, grab/lineman 0.321. `net = raw Г— (1 в€’ rate)`. (`lib/gp.js`)
- `sales_daily.net_revenue` = kshop + cash + delivery(after GP). Pastry & free-cup cost stored separately.
- Expense VAT 7%: `total = price Г— 1.07 Г— qty` when VAT on (baked into `total_amount`, not stored separately).
- OPEX rent WHT: pay owner 95%, withhold 5% (remit to Revenue Dept). Stored amount = full rent.
- OPEX VAT-auto suggestion = month net sales Г— 7%.
- Payroll (`payslip()` in `app/opex/opex-form.js`): SSO 5% of min(salary,15000) for employee AND company; commission = income Г— rate; commission tax 3%; gross = salary+position+commission+diligence; net transfer = salaryв€’ssoEmp+position+diligence+(commв€’commTax); **saved OPEX amount = companyCost = gross + companySSO**. **(2026-07-20) The per-employee OPEX amount is no longer a manually-typed field** вЂ” it used to require opening "аё„аёіаё™аё§аё“а№Ђаё‡аёґаё™а№Ђаё”аё·аё­аё™" and clicking a separate "а№ѓаёЉа№‰аёўаё­аё”аё™аёµа№‰" button to copy `companyCost` in, which was easy to forget (an employee's cost could silently stay at a stale/в‚ї0 figure for a whole month). It's now computed live from `payslip(e, income).companyCost` on every render and at submit time вЂ” there is no independent `amount` state for employees anymore, so editing salary/position/commission/diligence updates the displayed and saved OPEX figure immediately with no extra step.
- Employee defaults (`DEFAULT_EMPLOYEES`, `lib/opex.js`): emp1 13000/1500, emp2 12000/0; salary_dir default 36000. Payslip inputs persist in `localStorage` (`mm69_emp_slip`).

## Design system (2026-07-20 redesign вЂ” soft/modern, sidebar layout)

**Everything is driven by CSS custom properties defined once at the top of `app/globals.css` (`:root`).** To retheme the whole app, edit values there only вЂ” never hardcode a color/radius/shadow in a component.

- **Brand tokens:** `--color-primary` (coffee, primary actions/active states), `--color-accent` (taupe, highlights/focus). Swap these two to rebrand entirely.
- **Neutrals:** `--color-bg` (page canvas), `--color-surface`/`--color-surface-2`, `--color-border`, `--color-text`/`--color-text-muted`.
- **Radius scale:** `--radius-sm` (8px) / `--radius-md` (12px, default for cards/inputs/buttons) / `--radius-lg` (16px, cards/login) / `--radius-full` (pills/badges/chips). This is *the* lever for "soft vs boxy" вЂ” the previous Swiss-style redesign used a flat 2px everywhere, which read as rigid; this pass replaced it app-wide.
- **Shadow scale:** `--shadow-sm`/`--shadow-md`/`--shadow-lg` вЂ” soft, layered, multi-stop shadows instead of hard 1px borders for depth.
- **Legacy aliases:** `--coffee`, `--taupe`, `--border`, `--surface`, etc. all map onto the tokens above, so the many components using inline `style={{ ...: 'var(--coffee)' }}` continue to work without edits.
- **Inline-style radius:** components still use inline `style={{}}` objects (not just `className`) for layout-specific spacing вЂ” but every `borderRadius` value in those objects is `'var(--radius-md)'` (or `--radius-full` for true pills), never a hardcoded number. Grep for `borderRadius: [0-9]` before adding new inline styles вЂ” if you find a literal number, that's a bug, use a token.

**Layout вЂ” sidebar shell, not the old top-tab bar:**
- `components/app-shell.js` (client component вЂ” needs `mobileOpen` state + `usePathname`) renders `<div class="shell"><Sidebar .../><div class="shell-main"><div class="mobile-topbar">...</div><div class="desktop-topbar">...</div><div class="shell-container">{children}</div></div></div>`.
- **вљ пёЏ Gotcha (2026-07-20): `.mobile-topbar` must be an actual DOM child of `.shell-main`, never a flex sibling of `.sidebar` inside `.shell`.** An earlier version rendered it as a top-level sibling returned by `Sidebar` (so it became a flex item of `.shell`, a row-flex container, alongside `.shell-main`). On real iPhone Safari this caused `.mobile-topbar` to shrink-to-fit instead of taking its own row (tried `flex-wrap:wrap` + `width:100%` first вЂ” spec-correct but Safari didn't reliably honor it), squeezing `.shell-main` into a sliver and wrapping KPI numbers one digit per line. Fixed by moving `.mobile-topbar` to be a literal child of `.shell-main` (a column-flex container, so `align-items:stretch` gives it full width unconditionally вЂ” no flex-wrap dependency at all). **`mobileOpen` state now lives in `AppShell`** (passed down to `Sidebar` as `mobileOpen`/`setMobileOpen` props) since both the hamburger button (in `.mobile-topbar`, inside `.shell-main`) and the drawer/overlay (in `Sidebar`, a `.shell` sibling of `.shell-main`) need to share it, and they're no longer in the same DOM subtree.
- `components/sidebar.js` (client) renders `lib/perms.js` `NAV_TABS` (same source of truth as before) as a **collapsible left rail on desktop** (icon-only toggle, state remembered in `localStorage` key `mm69_sidebar_collapsed` вЂ” pure UI preference, not business data, so this doesn't violate the localStorage rule elsewhere in this doc) and a **slide-in drawer on mobile** (`<960px`, hamburger trigger + backdrop overlay, auto-closes on route change). Only renders the overlay + `<aside>` now вЂ” no longer owns `mobileOpen` (see above) or renders `.mobile-topbar` itself.
- `.desktop-topbar` (в‰Ґ961px only) shows the logged-in user's name/role вЂ” the one piece of session context that's otherwise hidden when the sidebar is collapsed to icon-only (`.sidebar-brand-text` hides in that state). Real session data only (`name`/`role` props `AppShell` already receives) вЂ” no fabricated notification bell/badge; don't add one without an actual notification data source behind it.
- `.wrap` class still exists (aliased to the same container styles) for anything not going through `AppShell`.

**KPI tiles вЂ” `components/kpi.js`:** icon-in-a-circle on the left, label/value/sub stacked on the right (`<Kpi icon="ti-xxx" label="..." value="..." sub="..." cls="green|red|blue" plain />`). Single shared component used by `dashboard`, `reports`, `analytics` вЂ” previously 3 near-duplicate inline implementations (one of which formatted money *inside* the component, two of which expected pre-formatted strings). All callers now pre-format via `fmtMoney()` before passing `value`. **`.kpis` uses fixed column counts per breakpoint (1/2/3), not `auto-fit`** вЂ” `auto-fit` packs as many columns as physically fit a wide desktop screen (e.g. 5), so a page with a tile count that isn't a clean multiple of that (dashboard has 6) left a dangling last row with several empty column-widths of blank space beside the orphaned tile. Fixed columns mean tile counts always wrap into clean, evenly-filled rows regardless of screen width.

**Date/month fields вЂ” `components/date-field.js`, mandatory for any new date or month picker:**
- **вљ пёЏ Gotcha: iOS Safari (WebKit) does not allow full CSS control over `input[type=date]`/`input[type=month]` internals.** Setting `background`/`color` works, but a native hairline/chrome inside the control still renders and can't be removed вЂ” not via `overflow:hidden` on the input (native widget paint mostly ignores its own `overflow`), not via a wrapping `overflow:hidden` div either (fixes gross bleed/overlap onto neighboring fields, but doesn't fix the internal hairline).
- `DateField` sidesteps this rather than fighting it: the real `<input type="date"|"month">` is still there and still fully drives interaction (opacity:0, absolutely positioned over a plain `<span>` that we fully control and style to match every other input exactly). Props: `value`, `onChange(valueString)` (not an event вЂ” different from a raw `<input onChange>`), `type="date"|"month"`, `min`, `max`, `disabled`, `placeholder`.
- **вљ пёЏ Gotcha: this overlay technique broke picker-opening on desktop while working fine on mobile.** iOS opens its native wheel picker on a tap anywhere in the input; desktop Chrome/Edge/Firefox only open the dropdown when the click lands on the native input's own small calendar-icon hotspot (clicking the text area just places a keyboard cursor) вЂ” hiding the whole input at `opacity:0` hides that hotspot with no visual cue for where it is, so desktop users had no way to open the picker at all. Fixed by calling `inputRef.current.showPicker()` on click of the wrapper (any click position), which works cross-browser wherever `showPicker()` is supported (falls back to native click-through behavior where it isn't).
- Used by all 7 date/month fields in the app: sales, expenses, analytics range picker, opex month + Form 50 date, reports month picker, settings data-tools. **Do not add a raw `<input type="date">`/`type="month">` anywhere вЂ” use `DateField`.**

**вљ пёЏ Gotcha (2026-07-20): forms that pre-fill from a date/month-scoped `existing` prop need `key={date}` (or `key={monthInput}`) on the component, not just the prop.** `SalesForm`/`OpexForm` initialize their fields via `useState(() => ...existing)`, which only runs on first mount. Changing the date/month via the URL re-fetches `existing` server-side and passes new props down, but React doesn't remount the client component just because props changed (same position in the tree) вЂ” so the *old* date's field values stayed frozen on screen while the newly-fetched data was silently ignored (reported as "picking a past date shows a blank form even though I already entered data for it"). `key={date}`/`key={monthInput}` on `<SalesForm>`/`<OpexForm>` in `page.js` forces a clean remount (and full state reset) on every date/month change вЂ” the standard React fix for this exact class of bug. `Form50` doesn't need this: its own state (payee info) is intentionally month-independent, and `amounts` are read reactively from props each render rather than copied into state.

**Responsive tables вЂ” `components/data-table.js`:**
- Desktop: renders a normal `<table>`. Mobile (`<720px`): pure-CSS transform (`.rtable` rules in globals.css) turns each row into a bordered, shadowed card with `data-label` pseudo-labels вЂ” no JS breakpoint logic.
- Usage: `<DataTable columns={[{key,label,align,render?}]} rows={[...]} rowKey={(r) => ...} />`. Applied to `/reports` (category breakdown) and `/analytics` (top materials, monthly comparison) as the reference implementations. **Other raw `<table>` usages in the app (e.g. `/admin/audit` inline diff table) were left as-is** (horizontal-scroll on mobile, still functional) вЂ” convert them to `DataTable` opportunistically when touching those files, not urgent.

**вљ пёЏ Gotcha (2026-07-19, `app/layout.js`): a custom `export const viewport` object REPLACES Next.js's default viewport meta tag entirely вЂ” it does not merge.** Omitting `width`/`initialScale` silently breaks mobile rendering (iOS Safari falls back to the ~980px legacy desktop viewport, shrinking the whole page). Always include `width: 'device-width', initialScale: 1` alongside any custom `themeColor` etc.
